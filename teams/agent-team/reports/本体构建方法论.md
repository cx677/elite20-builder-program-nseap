## 第一部分：本体是什么

### 1.1 一句话定义

> **本体（Ontology）= 企业认知操作系统（Enterprise Cognitive OS）= AI 理解世界、行动、学习的共享世界模型**

本体回答五个根本问题：

| 问题 | 含义 | 示例 |
|---|---|---|
| 什么存在 | Entity 实体 | Customer / Product / Order |
| 什么有关联 | Relation 关系 | Customer owns Asset |
| 什么可以发生 | Event 事件 | Create Order / Approve Contract |
| 什么可以执行 | Action 动作 | Submit / Review / Notify |
| 什么必须遵守 | Constraint 约束 | "Purchase Order must be approved before execution" |

---

### 1.2 十大核心认知（Richard 原文）

来源：`ontology-primer.docx` — AI 原生本体的十大核心认知

#### 认知 1：本体不是知识库，是企业认知操作系统

很多人把本体理解成数据字典、知识图谱、业务术语表，实际上都不对。

本体本质上是：**企业如何理解世界、理解业务、理解行动的认知模型**。

因此：**本体 = 企业认知操作系统（Enterprise Cognitive OS）**。

#### 认知 2：本体是 AI 的出厂说明书

大模型的问题不是不聪明，而是**不知道这家公司怎么运作**。

> 同样一句"帮我处理一个紧急订单"，不同企业含义完全不同：
> - 有的：优先客户等级
> - 有的：优先利润
> - 有的：优先交付日期
> - 有的：优先战略客户

这些规则并不在大模型参数里，而是在企业本体里。

**因此：本体是 AI 进入企业后的出厂配置文件。**

#### 认知 3：本体是从语言理解到业务执行的桥梁

LLM 天然擅长语言、对话、推理，但企业需要的是执行。

> 客户说"我投诉了"——
> - LLM 理解没问题
> - 但企业需要：创建工单 → 通知经理 → 调取历史记录 → 判断 SLA → 升级处理

这就需要：**语言 → 本体 → 动作** 转换链路。

**因此：本体是 AI 从"能说"走向"能干"的桥梁。**

#### 认知 4：本体是企业经验的数字化载体

企业最有价值的资产往往不在系统里，而在：
- 老员工脑子里
- Excel 里
- 微信群里
- SOP 里
- Shadow System 里

> 老师傅知道：哪种插单必须拒绝、哪种插单可以接受、哪种插单需要先备料。但没人写下来。

本体工程的核心任务：**把隐性经验显性化**，形成 `经验 → 规则 → 本体 → Agent 能力` 闭环。

#### 认知 5：本体是 Agent 的世界模型

未来企业不是使用一个 AI，而是**数百个 Agent**：
- 销售 Agent
- 财务 Agent
- HR Agent
- 制造 Agent
- 风控 Agent

如果没有统一本体，每个 Agent 都有自己的理解，最终形成 **AI 孤岛**。

**因此：本体是所有 Agent 共享的世界模型（World Model）。**

#### 认知 6：本体比模型更稳定

| 资产 | 变化频率 | 长期价值 |
|---|---|---|
| 大模型 | 每月变（GPT、Claude、DeepSeek、Gemini） | 短期执行引擎 |
| **企业核心业务** | 十年不变（BOM、客户生命周期、风控规则、生产流程）| **长期资产** |

**因此：企业长期资产不是模型，而是 `本体 + 数据 + Skill`。模型只是执行引擎。**

#### 认知 7：本体是 Skill Factory 的基础

未来企业竞争不是模型竞争，而是 **Skill 竞争**：
- 采购 Skill
- 报价 Skill
- 招投标 Skill
- 风控 Skill
- 客服 Skill

这些 Skill 都依赖：
- Input Schema
- Output Schema
- 实体定义
- 流程定义
- 规则定义

而这些**全部来自本体**。

**因此：Skill Factory 的原材料就是 Ontology。**

#### 认知 8：本体是企业系统的通用语言

今天企业系统碎片化严重：ERP / CRM / MES / PLM / OA / WMS。系统之间大量成本浪费在**字段映射**上。

| 系统 | 字段 | 实际意义 |
|---|---|---|
| CRM | `Customer_ID` | Customer |
| ERP | `Client_No` | Customer |
| MES | `Account_Number` | Customer |

如果有统一本体，所有系统直接映射到 **Customer Ontology** 即可。

**因此：本体成为企业数字世界的统一语言层。**

#### 认知 9：本体是持续学习的基础

在 KSTAR 框架里：

```
K = Knowledge
   = Ontology + Skill + Meta-Skill
```

| 组成部分 | 决定什么 |
|---|---|
| **Ontology** | Agent 如何理解世界 |
| **Skill** | Agent 如何行动 |
| **Meta-Skill** | Agent 如何改进自己 |

Agent 进化的第一层基础设施就是 **Ontology**。没有本体，学习结果无法沉淀。

#### 认知 10：AI 落地最大的瓶颈是本体工程

今天多数 AI 项目失败原因：不是模型、不是算力、不是 GPU，而是**业务没有被结构化**。

企业自己说不清：
- 业务对象是什么
- 流程是什么
- 决策依据是什么
- 例外情况是什么

**结果：AI 只能聊天，不能工作。**

### 1.3 三层结构：T-Box / A-Box / R-Box

来源：英文 PPT `Ontology, Neurosymbolic AI, and Agentic Reasoning` slide 6/7

```text
┌──────────────────────────────────────────────────┐
│  T-BOX  (Terminology)                            │
│  ─────                                            │
│  概念 / 类别 / 术语层                              │
│  Examples: Customer, Order, Product,              │
│            Complaint, Equipment                   │
│  → 词汇层，定义"有什么"                            │
└──────────────────────────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────┐
│  A-BOX  (Assertions)                             │
│  ─────                                            │
│  实例 / 事实 / 断言层                              │
│  Examples:                                        │
│    Customer #1024 购买了 Order #567               │
│    Equipment #88 当前在 Sensor #12 监测下         │
│  → 描述"当前为真的事实"                            │
└──────────────────────────────────────────────────┘
                        ↓
┌──────────────────────────────────────────────────┐
│  R-BOX  (Rules / Relations)                      │
│  ─────                                            │
│  规则 / 关系 / 约束层                              │
│  Examples:                                        │
│    Rules / Constraints / Inference / Causality    │
│  → 通往可行动认知的桥梁                            │
└──────────────────────────────────────────────────┘
```

**关键洞察（slide 7）**：

> 智能在关系里（Intelligence lives in the relationships），不在实体里。R-Box 是神经符号 AI 的关键。

### 方法一：Ontology Factory 7 步流水线（工具层）

来源：`Ontology-Factory-Base.md` §3.2 + `Ontology-Factory-HyperExtract.md` §5

#### 适用场景

- 企业级大规模本体工厂
- 输入：4 类源（结构化/非结构化/代码/知识工程）
- 输出：完整 Ontology Package

#### 4 类输入源

| 类别 | 示例 | 特点 |
|---|---|---|
| **非结构化语义源** | 论文 / 规范 / 标准 / 手册 / SOP / 合同 / 报告 / 演示文稿 / 图表 / 扫描件 | 传统为人读写的语义块 |
| **结构化数据源** | 元数据 / 数据字典 / 数据仓库 schema / SQL 关系库 / ERP/MES/PLM schema / 表 / 列 / 外键 | 已结构化但缺显式语义 |
| **代码库源** | C/C++ .h 头文件 / API 定义 / 类型定义 / 接口文件 / ORM 模型 / Headless CMS / 配置文件 / 协议定义 | 真实运行模型的源头 |
| **知识工程源** | 图数据库 / RDF/OWL 本体 / Wikidata / Wikipedia / 现有 KG / 分类法 / 受控词表 / 企业知识库 | 已含部分知识结构，可导入对齐 |

#### 7 步流水线（Pipeline Type 1：非结构化源）

```text
Unstructured Source（PDF/Word/PPT/MD/SOP/合同/标准/图片）
  ↓
① Document Parsing（文档解析）
  ↓
② Semantic Block Extraction（语义块抽取）
  ↓
③ Entity / Concept Extraction（实体/概念抽取）  → AutoConcept
  ↓
④ Relation Extraction（关系抽取）                → AutoRelation
  ↓
⑤ Rule / Constraint Extraction（规则/约束抽取）  → AutoConstraint → R-Box
  ↓
⑥ Event / Process Extraction（事件/流程抽取）    → AutoEvent / AutoWorkflow
  ↓
⑦ Ontology Assembly（本体装配）
  ↓
Validation（验证）
  ↓
Ontology Package 治理包
```

#### 8 种自动抽取能力

| 能力 | 抽取内容 | 映射到 Box | 示例 |
|---|---|---|---|
| **AutoConcept** | 概念抽取 | T-Box | Customer / Supplier / Product / Asset / Order / Task / Agent |
| **AutoTaxonomy** | 分类体系 | T-Box | Asset → Equipment / Robot / Sensor |
| **AutoRelation** | 关系抽取 | T-Box+关系 | Customer owns Asset / Task uses Skill / Agent executes Task |
| **AutoConstraint** | 约束规则 | **R-Box** | "Purchase Order must be approved before execution" |
| **AutoEvent** | 事件本体 | Event Model | Create Order / Approve Contract / Start Production |
| **AutoWorkflow** | 流程本体 | Process Model | Task Flow / BPMN |
| **AutoSkill** | 技能本体 | Skill Candidates | Contract Review / Bid Evaluation / Risk Assessment |
| **AutoAgent** | Agent 角色 | Agent Role Candidates | Sales Agent / Procurement Agent / Planner Agent / Companion Agent |

#### 输出：Ontology Package（不是单纯知识图谱）

```text
Ontology Package
├── T-Box        概念、类、属性、继承、约束
├── A-Box        实例与示例
├── R-Box        规则、约束、策略
├── Process Model
├── Event Model
├── Skill Candidates
├── Agent Role Candidates
└── Validation Report
```

#### 8 大核心能力（Hyper-Extract v0.9 §5）

| 能力 | 关键内容 |
|---|---|
| §5.1 Ontology Extraction | 8 种 Auto* 抽取 |
| §5.2 Ontology Engineering | T-Box Designer / A-Box Manager / R-Box Designer / Visual Ontology Editor（类 Protégé）|
| §5.3 Agent Ontology Layer | 新增 Agent/Skill/Goal/Role/Memory/Tool/Policy；**自动转 IEEE P3394 / MCP / A2A / OpenAPI** |
| §5.4 Skill Ontology Factory | Skill 必有 Input/Output/Evaluation Rubric/Action Plan/Constraints/Learning Signals |
| §5.5 Enterprise Ontology Templates | 5 个行业：Manufacturing(ISA95/88/OPC-UA) / Energy / Education(P3428) / Healthcare(FHIR) / Finance |
| §5.6 **Ontology Evolution Engine** | **Git for Ontology**：Versioning / Branching / Merging / Diff + Ontology PR/Review/Release |
| §5.7 KSTAR Integration | 每个 KSTAR Episode 自动产生 New Concept/Rule/Skill/Agent Pattern |
| §5.8 Ontology MCP Server | 暴露 Search/Query/Update/Generate Skill/Generate Agent 5 个服务 |

#### 在 ECS 中的角色

```text
Enterprise Sources
      ↓
Ontology Factory
      ↓
Enterprise Ontology
      ↓
Skills
      ↓
Agents
      ↓
KSTAR Episodes
      ↓
Enterprise Cognitive Assets
```

支持：Agent 推理 / Skill 组合 / 工作流自动化 / 知识治理 / 认知记忆 / 企业学习循环。

### 方法二：AI 需求 4 阶段法（方法层）

来源：`Agent-system-requirements-as-cognitive-task.docx`

#### 适用场景

- 单个 AI/Agent 项目的认知设计
- 从"业务方要 AI 助手"翻译成"具体认知任务"
- 顶层方法论，决定"做什么"

#### 4 阶段总览

```text
需求分析 → 认知设计 → 方案工程 → 反馈进化
```

> AI 项目最大的挑战不是编码，而是：如何把业务问题转化为机器可以理解、推理、执行和持续优化的认知能力。

---

#### 阶段 1：需求分析（Requirement Analysis）

**目标**：从业务语言中识别真实问题。

大部分业务需求实际上都是**"解决方案伪装成需求"**：

| 错误的需求 | 翻译为真实问题 |
|---|---|
| "做一个 AI 助手" | ❌ 这不是需求 |
| "做一个智能体" | ❌ 这不是需求 |
| "做一个知识库" | ❌ 这不是需求 |
| "做一个自动报告系统" | ❌ 这不是需求 |
| "做一个聊天机器人" | ❌ 这不是需求 |

**步骤 1：需求净化**——持续追问 3 个问题：

```text
发生了什么？         → 现象（Symptoms）
为什么重要？         → 业务影响（Business Impact）
不解决会怎样？       → 业务后果（Business Consequence）
```

**示例**：

| 问题 | 答案 |
|---|---|
| 发生了什么？ | 客户流失率上升 |
| 为什么重要？ | 月收入下降 15% |
| 不解决会怎样？ | 未来半年可能损失 3000 万收入 |

**步骤 2：认知任务识别**——把业务问题翻译为真正的认知任务：

| 业务问题 | 本质认知任务 |
|---|---|
| 客户流失 | **预测** |
| 工单太多 | **分类** |
| 找不到资料 | **检索** |
| 专家经验难复制 | **推理** |
| 排产困难 | **优化** |
| 风险控制 | **决策支持** |

**输出**：认知任务定义（Cognitive Task Definition）

> ✅ "预测客户未来 30 天流失概率"
> ❌ "做一个客户分析 Agent"

---

#### 阶段 2：认知设计（Cognitive Design）—— 整个 AI 项目最核心

**目标**：定义完成任务所需的知识结构。

```text
第 1 层：概念建模（Ontology / T-Box）
        回答：系统需要理解什么？
        产出：Customer / Product / Order / Contract / Campaign / Channel

第 2 层：事实建模（A-Box）
        回答：系统需要掌握哪些事实？
        产出：客户A 购买 产品B

第 3 层：关系建模（R-Box）
        回答：这些对象之间如何关联？
        产出：购买行为影响客户价值

第 4 层：逻辑建模（专家规则 + 因果 + 决策）
        回答：专家如何判断？
        产出："连续 30 天活跃下降 + 投诉增加 → 高风险流失"
```

**输出**：完整认知模型（Cognitive Model）

---

#### 阶段 3：方案工程（Solution Engineering）

**目标**：将认知模型转化为可执行系统。

```text
第 1 层：能力映射
        感知 / 检索 / 推理 / 规划 / 执行 / 学习

第 2 层：Skill 设计
        定义：输入 → 处理 → 输出
        例：客户行为数据 → 流失预测 Skill → 风险评分

第 3 层：Agent 设计
        多 Skill 组合形成 Agent
        例：数据分析 + 预测 + 推荐 + 解释 = 客户运营 Agent

第 4 层：系统架构
        Prompt / Workflow / Agent / Fine-tuning / KG / Rules Engine / Simulation
```

**输出形式**：API / Chat / Dashboard / Copilot / Autonomous Agent

---

#### 阶段 4：反馈进化（Feedback Evolution）

```text
执行结果
  ↓
评估
  ↓
学习信号（ΔR）
  ↓
反馈到 Ontology / Skill / Meta-Skill
```

### 方法三：Skill-Ontology 4 周法（教学层）

来源：`Skill-ontology-course-idea.docx`

#### 适用场景

- 教学/培训（4 周让学生从 0 到交付 Agent 系统）
- 团队快速复用统一方法
- 教学化方法，让团队"会做"

#### 核心主线

```text
Ontology → Meta-Skill → Skill → Task Agent → Agent Team → Evaluation → KSTAR Loop → Delivery
```

#### 4 周教学大纲

| 周 | 主题 | 模块 1 | 模块 2 | 模块 3 | 模块 4 | 周产出 |
|---|---|---|---|---|---|---|
| **W1** | Skill-Ontology 基础 | 范式转换（App → Skill）| 四层 Agent Memory | Ontology Snapshot | Meta-Skill 模板 | Skill Delivery Map / Memory Blueprint / Task Ontology Snapshot / Meta-Skill Set |
| **W2** | Skill 封装与运行 | Skill 封装 | 语义触发 | Hooks 质量门禁 | MCP 自动化管道 | Skill Package V1 / Semantic Router / QA Hooks / Automated Pipeline |
| **W3** | AG2 Agent Team + KSTAR | AG2 Beta | Built-in Skills | Orchestration 编排 | KSTAR Evaluation | AG2 Team / Agent App V1 / Orchestration Plan / KSTAR Pipeline |
| **W4** | OpenClaw 部署进化 | Agent Runtime | 多平台分发 | Telemetry 监控 | 持续进化 | Runtime Integration / Multi-Channel / Eval Dashboard / Self-Improving System |

#### 4 周收获

**W1 收获**：掌握轻量级 Agent 交付的基础结构

```
Ontology Snapshot + Meta-Skill Template = 可实例化 Skill
```

**W2 收获**：掌握把一次性任务转化为可复用、可运行、可监控的 Skill

```
Skill 封装 → Skill 触发 → Ontology 绑定 → Hook 校验 → 自动化执行
```

**W3 收获**：掌握从 Skill 到 Agent Team 的升级路径

```
Skill → Agent Role → Agent Team → Orchestration → Evaluation → ΔR Learning Signal
```

**W4 收获**：完成从原型到产品的闭环

```
Agent App → Runtime → Service Delivery → Telemetry → Evaluation → KSTAR Learning → Skill/Ontology Evolution
```

### 方法四：Ontology Broker 结构洞法（治理层）

来源：`Idea-on-ontology-broker.docx`，Richard 原创

#### 适用场景

- 长期本体治理
- 发现"应该创建但还没创建"的本体节点
- 让本体从"静态文档"变成"活体"

#### 核心思想

> 把社会学 Burt 的"**结构洞理论**"反向应用——
>
> 不是问"**人之间的经纪人是谁**"，
> 而是问"**应该创建哪些本体概念来桥接断开的认知区域**"。

#### 9 步核心流程

```text
观察（Observe usage）
  ↓
观察语义流量（Observe semantic traffic）
  ↓
发现结构洞（Find structure holes）
  ↓
提出本体扩展（Propose ontology extensions）
  ↓
评估（Evaluate）
  ↓
接受/拒绝（Accept or reject）
  ↓
本体进化（Ontology evolves）
```

#### 9 大机制

| 机制 | 描述 | 例子 |
|---|---|---|
| 1. 把本体视为语义图谱 | 每个节点是 Entity / Skill / Agent / Process / Event / Capability / Policy / Memory | ISA95 Equipment / SAP Material / MES Order 都在图里 |
| 2. 每个 Agent 住在子图 | Maintenance / Production / Quality 各自密集内部连接 | 但跨域连接稀疏 |
| 3. 结构洞 = 缺失的语义桥梁 | Maintenance 与 Production 通过 Equipment 间接连接 | 但**没有 EquipmentFailureImpact 节点** |
| 4. Broker 自动填补 | Agent A 反复问"对生产有什么影响" | Broker 提议新增 `EquipmentFailureImpact` 或 `ProductionRisk` 节点 |
| 5. 结构洞分数 = 进化优先级 | 节点有 frequency / traffic / importance | 高 traffic + 低 link = 高优先级 |
| 6. 发现可复用 Skill | Scheduling / Maintenance / Supply 三个 Agent 反复调用类似推理 | Broker 引入 **Risk Forecast Skill** 共享 |
| 7. 发现新 Task Agent | Maintenance / Production / Quality 频繁通信 | Broker 推荐 **Manufacturing Coordination Agent** |
| 8. 个性化 Companion Agent | 工程师 A 在 PLC/Robot/Sensor，经理 B 在 Cost/Schedule/Risk | Companion 主动学习用户最关心的"结构洞"区域 |
| 9. 动态本体进化 | 传统："先设计本体" → NSEAP 哲学："**观察使用 → 提议 → 评估 → 接受**" | 本体变成活体 |

#### 与神经符号 AI 的整合

```text
Interaction Logs
  ↓
Semantic Traffic Graph
  ↓
Structure Hole Detection
  ↓
Candidate Ontology Nodes
  ↓
LLM Proposal
  ↓
Ontology Validation
```

#### 传统 vs NSEAP 哲学

| 传统本体工程 | NSEAP 哲学 |
|---|---|
| 先设计本体 | **观察使用** |
| | ↓ |
| | 观察语义流量 |
| | ↓ |
| | 发现结构洞 |
| | ↓ |
| | 提议本体扩展 |
| | ↓ |
| | 评估 |
| | ↓ |
| | 接受或拒绝 |
| | ↓ |
| | 本体进化 |

**结果：本体成为活体（living artifact）**。

## 第四部分：OKF 落盘格式

来源：`Ontology-format-Okf-by-Google.md` — Google 开源的 Open Knowledge Format

### 4.1 一句话定义

> **OKF（Open Knowledge Format）= 用 Markdown 目录组织知识 + YAML Frontmatter 做元数据 + 标准 Markdown 链接形成知识图谱的标准格式**

### 4.2 Bundle 目录结构

```text
my-knowledge-bundle/
├── index.md              # 目录索引（渐进式浏览）
├── log.md                # 变更历史
├── tables/               # 按主题组织
│   ├── index.md
│   ├── orders.md
│   └── customers.md
├── metrics/
│   └── weekly_active_users.md
└── ...
```

### 4.3 单个 Concept 文件结构（最重要）

```markdown
---
type: BigQuery Table              # 【必填】概念类型，自由命名
title: Customer Orders            # 推荐：显示名
description: 一行总结描述           # 推荐：摘要
resource: https://...             # 推荐：原始资源 URI
tags: [sales, revenue, orders]    # 推荐：标签
timestamp: 2026-06-01T12:00:00Z   # 推荐：最后修改时间
---

# Schema
| Column | Type | Description |
|---|---|---|
| order_id | STRING | ... |

# Joins
与 [/tables/customers.md] 关联...

# Citations
[1] 来源链接...
```

### 4.4 链接与知识图谱

- 使用标准 Markdown 链接
- 推荐绝对路径（从 bundle 根开始）：`[/tables/customers.md]`
- 相对路径也可：`[./other.md]`
- 这些链接让整个 bundle 成为**可遍历的图谱**，Agent 可以跟随关系

### 4.5 OKF 与本体的映射

| OKF 概念 | 本体概念 | 关系 |
|---|---|---|
| Knowledge Bundle | Ontology Package | OKF 是物理载体，本体是逻辑结构 |
| Concept (.md 文件) | T-Box / A-Box | 一个文件可承载类、实例、规则 |
| Frontmatter | Manifest / Schema | 结构化元数据 |
| Markdown 链接 | T-Box 关系 | 关系用链接表达，自动形成图谱 |
| index.md | Ontology Index | 入口和导航 |
| log.md | Ontology Versioning | 变更历史 |

**OKF 是"本体的 GitHub 化"** —— 用最简单的方式（Markdown）让本体可分发、可版本化、可被 Agent 解析。

### 4.6 实际使用步骤

1. **手动创建**：用任意文本编辑器创建目录和 .md 文件，放入 Git 仓库
2. **自动生成**：Google 提供了 Reference Agent，可从 BigQuery 等来源自动生成 bundle
3. **消费/读取**：
   - 人类：VS Code / Obsidian / GitHub
   - AI Agent：加载目录文件到上下文
   - 可视化：`viz.html` 交互式图谱
4. **分发与集成**：GitHub / GitLab / ZIP / 静态服务器

### 4.7 优势

- 极简：无需 SDK、数据库
- 互操作：跨平台、跨 Agent、跨组织
- 适合 AI 知识管理、Agent 技能本体、RAG

## 第五部分：神经符号 AI 范式

来源：英文 PPT `Ontology, Neurosymbolic AI, and Agentic Reasoning`

### 5.1 为什么需要 Ontology（Why Ontology Matters）

```text
DATA         → Raw signals, records
INFORMATION  → Organized, queryable
KNOWLEDGE    → Structured meaning
REASONING    → Inference, decisions, learning
```

**缺失的层**：现代 AI 仍在挣扎于

- **Context** — 在这种情境下是什么意思
- **Causality** — 为什么会发生
- **Consistency** — 同一案例，同一动作

### 5.2 范式转移

```text
OLD PARADIGM                              NEW PARADIGM
────────────                              ──────────────
Data Processing                           Cognitive Reasoning & Adaptive Action
数字化模拟流程                              理解意义（不仅是存储记录）
存储与检索                                  推理关系与后果
工作流自动化                                在现实世界采取行动
事务处理                                   从结果中持续学习
```

> 未来不是会生成语言的智能系统。

### 5.3 神经 vs 符号：各自的能力

| Neural / LLM | Symbolic / Ontology |
|---|---|
| 语言理解 | 结构化推理 |
| 语义解释 | 规则与约束 |
| 模式生成 | 关系逻辑 |
| 人机交互 | 可解释性 |

**合并**：Neurosymbolic AI = **fluency meets formality**（流畅性遇上形式化）。

### 5.4 4 步认知循环

```text
01 OBSERVE    观察    → What?      → "Shipment delayed."
02 INTERPRET  解释    → So What?   → "High-value customer may churn."
03 ACT        行动    → Now What?  → "Trigger retention workflow."
04 LEARN      学习    → What next? → "Outcome feeds next cycle."
```

### 5.5 交织认知循环

```text
Neural（神经）  →  Interpret（LLM resolves ambiguity）
Symbolic（符号）→  Structure（Ontology organizes meaning）
                →  Validate（Symbolic logic checks）
Action（行动）  →  Act（Agent executes the decision）
Feedback（反馈）→  Learn（Outcome feeds the next cycle）
```

**FEEDBACK — the loop closes**（反馈——循环闭合）。

### 5.6 5 步从演示到推理系统

1. **Learn ontology and graph thinking** — 把关系视为一等数据
2. **Develop agentic design skills** — Plan / Decide / Act / Learn
3. **Build neurosymbolic workflows** — LLM 流畅 + 符号结构
4. **Anchor in real business actionability** — 推理改变结果
5. **Adopt evaluation as infrastructure** — 评估即基础设施

### 5.7 两个真实故事

#### 故事一：情报分析

- **挑战**：海量碎片化情报数据，异构源不兼容格式，弱信号人手无法关联
- **本体方法**：
  - 映射实体：人、地点、通信、交易、事件
  - 定义跨数据集关系
  - 通过推理浮现异常和隐藏关联
- **结果**：实体图谱揭示 Subject / Courier / Compound 之间的隐藏网络

#### 故事二：Agentic 客服

- **场景**：愤怒客户、物流延迟、高忠诚度、多次历史投诉
- **脚本无法处理**：需要 Context + Urgency + Sentiment + Policy + History 组合判断
- **神经符号流**：
  - LLM 解释语言、意图、情感
  - 本体组织 customer / order / policy / history
  - 规则做约束校验
  - Agent 执行决策

## 第六部分：落地清单

### 6.1 启动一个本体项目时的步骤

```text
Step 1: 需求净化（方法 2 阶段 1）
        把"做 AI 助手"翻译成"预测客户未来 30 天流失概率"

Step 2: 认知设计（方法 2 阶段 2）
        T-Box：定义 Customer / Product / Order / Contract
        A-Box：识别 客户A 购买 产品B 的事实
        R-Box：定义 购买行为影响客户价值 的规则
        Logic：编码专家"连续 30 天活跃下降 + 投诉增加 → 高风险"

Step 3: 方案工程（方法 2 阶段 3）
        能力映射 → Skill 设计 → Agent 设计 → 系统架构

Step 4: 教学化（方法 3）
        4 周让团队"会做"：Ontology Snapshot → Meta-Skill → Skill → Task Agent → Agent Team → KSTAR

Step 5: 本体工厂化（方法 1）
        把已设计的本体接上 4 类源 + 7 步流水线
        输出标准 Ontology Package（T/A/R-Box + Process + Event + Skill + Agent）

Step 6: OKF 落盘
        每个 Concept 一个 .md 文件
        Frontmatter 做元数据
        Markdown 链接形成图谱
        index.md 导航 + log.md 版本化

Step 7: 治理进化（方法 4）
        持续观察 Agent 使用
        发现结构洞
        Broker 自动提议新节点 / 新 Skill
        评估后接受
        本体活起来
```

### 6.2 评估清单

本体是否做得"对"，可以问 10 个问题：

| # | 问题 | 对应认知 |
|---|---|---|
| 1 | 本体有没有"事实"层（A-Box），还是只有术语？ | #1 认知 OS |
| 2 | 业务专家看着本体能不能说出"这是我脑子里的规则"？ | #4 经验载体 |
| 3 | Agent 拿这个本体能不能行动？还是只能查询？ | #3 语言→动作 |
| 4 | 多个 Agent 用的是同一份本体吗？ | #5 世界模型 |
| 5 | 业务规则变化时，本体几天能更新？ | #6 比模型稳定 |
| 6 | 能不能用本体驱动 Skill 的 Input/Output Schema？ | #7 Skill Factory |
| 7 | 别的系统能不能直接映射到本体？ | #8 通用语言 |
| 8 | 每个 KSTAR Episode 跑完能不能沉淀回本体？ | #9 持续学习 |
| 9 | 业务人员自己能不能用本体说明业务？ | #10 结构化 |
| 10 | 本体有 R-Box 规则层吗？还是只有 T-Box 概念？ | 1.3 三层结构 |

### 6.3 常见错误

| 错误 | 后果 | 正确做法 |
|---|---|---|
| 把"做 AI 助手"当作需求 | 项目交付物不是真问题 | 用方法 2 阶段 1 做需求净化 |
| 只做 T-Box 概念 | 本体变成术语表 | 必做 A-Box 实例 + R-Box 规则 |
| 不让业务专家参与 | 本体偏离现实 | 业务专家共同定义 R-Box |
| 用 Excel 维护本体 | 半年后没人维护 | 用 OKF + Git 版本化 |
| 不接 Agent | 本体只是文档 | 必接 Skill + Agent，让本体"干活" |
| 没有学习闭环 | 本体不会进化 | 接 KSTAR Episode → Ontology 反馈 |
