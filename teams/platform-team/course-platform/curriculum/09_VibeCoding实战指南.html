📖 第1步 · 读 README — 理解核心理念

目标：建立对 Vibe Coding 的整体认知，理解它不是一个偷懒工具， 而是一套「规划驱动 + 上下文固定 + AI 结对执行」的工程化方法论。

一句话定义

Vibe Coding = 规划驱动 + 上下文固定 + AI 结对执行，让「从想法到可维护代码」变成一条可审计的流水线，而不是一团无法迭代的巨石文件。

核心信念（作者原话）

规划就是一切。谨慎让 AI 自主规划，否则你的代码库会变成一团无法管理的乱麻。

你能从这个仓库得到什么

  • 成体系的提示词工具链 — 系统提示词约束 AI 行为，编程提示词提供需求澄清→计划→执行的全链路脚本
  • 闭环交付路径 — 需求 → 上下文文档 → 实施计划 → 分步实现 → 自测 → 进度记录，全程可复盘可移交
  • 模块化技能体系 — 面向特定领域（数据库、API 集成、DevOps）的即用型 skills
  • 元方法论 — 递归自优化系统：用 AI 生成提示词 → 再用 AI 优化这些提示词 → 循环迭代

编码规范要点

  • 语言策略：文档/注释用中文，代码符号（函数/变量/模块）用英文
  • 缩进：空格缩进 2 或 4 可选但不得混用，行宽 ≤ 120 列
  • 命名：小写加中划线或下划线，不用空格和特殊字符
  • 函数：单一职责、短小精悍、接口签名稳定
  • 提交格式：feat|fix|docs|chore|refactor|test: scope – summary
  • 提交前清单：跑通 make lint、无临时文件、更新对应文档

📊

🔄

🎯

一次只做一件事，极致的专注可以击穿代码。

法 · 工作方法

原则说明
一句话目标 + 非目标明确做什么、更明确不做什么
正交性功能不要重复，分场景看待
能抄不写先问 AI 有没有合适仓库，下载下来改
先看官方文档把官方文档爬下来喂给 AI
按职责拆模块功能模块边界清晰
接口先行，实现后补先定义契约再实现细节
一次只改一个模块避免连锁故障
文档即上下文文档是写给 AI 看的，不是事后补的

术 · 执行技巧

  • 明确边界：告诉 AI 能改什么、不能改什么（文件/范围/风格）
  • Debug 模板：只给「预期 vs 实际 + 最小复现」，不给整段代码
  • 测试策略：AI 写测试，人来审断言
  • 会话管理：代码一多就切新会话，避免上下文污染
  • 重复尝试：多试几次，实在不行重开窗口

元方法论：递归自优化系统

该仓库最有价值的思想之一：

  1. 定义两个角色：α-提示词（生成器）和 Ω-提示词（优化器）
  2. 创生：用 AI 生成 α 和 Ω 的 v1 版本
  3. 自省与进化：用 Ω(v1) 优化 α(v1) → α(v2)
  4. 创造：用 α(v2) 生成所有需要的目标提示词
  5. 循环：将新生成的产物反馈回系统，启动持续进化
核心理念：通过持续的递归优化循环，系统在每次迭代中实现自我超越，无限逼近预设的预期状态。

⚡ 第4步 · 编程提示词 — 让 AI 高效产出

目标：学习并使用 i18n/zh/prompts/coding_prompts/ 中的实战提示词。 这些是被反复打磨过的"可执行脚本"，直接复制给 AI 就能产出高质量代码。

典型编程提示词模板

仓库中的 coding_prompts 通常遵循这个结构：

## 角色定位
你是一个 [领域] 专家，擅长 [具体能力]

## 约束条件
- 能改什么：[文件范围]
- 不能改什么：[边界限制]
- 风格要求：[编码规范]

## 任务目标
一句话描述要做什么

## 非目标
明确不要做的 N 件事

## 输入
- 现有代码/文件路径
- 参考文档/接口定义

## 输出要求
- 文件结构
- 代码质量门槛
- 验收标准

实战示例：需求澄清提示词

📝 一句话目标：明确需求的输入、输出和边界
非目标：不要开始写代码，不要讨论实现细节

请帮我澄清这个需求：
1. 输入是什么？从哪来？
2. 输出是什么？给谁用？
3. 成功标准是什么？
4. 什么情况不属于这个需求范围？
5. 有没有类似实现可以参考？

实战示例：实施计划提示词

📝 一句话目标：制定可执行的分步实施计划
非目标：不要写代码，不要做技术选型假设

基于以下需求，制定实施计划：
1. 需要创建/修改哪些文件？
2. 执行顺序是什么？依赖关系？
3. 每一步的验收标准？
4. 预计风险点在哪？
5. 哪些可以并行？

🔒 第5步 · 系统提示词 — 约束 AI 行为边界

目标：学习如何写高质量的 system prompt 来驯服 AI。 这是控制 AI 产出质量最关键的能力。

为什么系统提示词如此重要？

AI 模型默认是无边界状态的——它可以自由选择任何方向。 系统提示词的作用不是"让 AI 更聪明"，而是精确划定它的行为范围。 作者在仓库中收录了多个版本的 system prompts，对应不同的开发场景。

系统提示词的核心要素

要素说明
角色定义你是谁？擅长什么？局限在哪？
行为边界能做什么、不能做什么（红线）
输出规范格式、风格、语言、长度限制
推理指南先分析再回答，还是直接给方案
知识范围使用哪些上下文、依赖哪些资源
反馈机制不确定时怎么做

系统提示词构建原则（来自仓库文档）

  • 明确性优先：模糊的指令→模糊的输出。每条指令应该只有一种理解方式
  • 结构化呈现：用层级、列表、代码块组织提示词，而不是长篇段落
  • 行为示例：给出好/坏输出的例子，让 AI 知道哪种是你想要的
  • 渐进约束：从宽松到严格逐步限制，不要一次性把所有规则压上
  • 可测试性：每条约束应该能通过输出验证是否被遵守
  • 持续优化：用 Ω-提示词来优化 α-提示词，递归改进

仓库中的系统提示词版本

i18n/zh/prompts/system_prompts/ 包含多个版本的开发规范与思维框架， 每个版本对应不同的使用场景（独立开发、团队协作、复杂项目等）。 它们不仅是提示词，更是可复用的行为契约。

🧩 第6步 · Skills — 模块化技能的构建与复用

目标：理解模块化技能的设计哲学，学会把领域知识封装成 AI 可复用的 skill。 这与模块化 Skill 系统的设计思路完全一致。

什么是 Skill？

Skill（技能）是一份结构化的 markdown 文档，包含特定领域的完整知识： 概念说明、API 文档、常用命令、典型工作流、陷阱与规避。 当 AI Agent 加载一个 skill 后，它在处理该领域问题时就拥有了专家级上下文。

🤖

Claude Code

安装、配置、常用命令链、MCP集成

🗄️

PostgreSQL

查询优化、索引策略、事务隔离级别、备份恢复

📱

Telegram Dev

Bot API、Webhook设置、消息类型、支付集成

Skill 的标准结构

# 技能名称

## 概述
一句话说明这个技能解决什么问题

## 前置条件
- 需要的工具/环境
- 依赖的版本要求

## 核心概念
- 领域术语解释
- 关键架构图

## 常用命令 / API
- 高频操作的命令和用法
- 带实际示例

## 工作流
- 典型场景的完整步骤
- 从入门到完成

## 陷阱与规避
- 常见的坑和解决方法
- 性能问题和安全注意事项

## 相关技能 / 参考链接

如何为自己的项目构建 Skill

  1. 识别领域：你的项目中哪些部分是重复出现的专业领域（数据库、API、部署等）
  2. 收集知识：官方文档、常用命令、过往踩坑记录
  3. 结构化输出：按上面的模板组织
  4. 测试验证：让 AI 加载 skill 后完成一个典型任务，看是否够用
  5. 持续维护：每次踩到新坑就补充到 skill 的陷阱部分

如何应用到自己的项目

将 Skill 体系与提示词工程结合，可以实现：

  • 多角色 Agent 协作：用系统提示词为每个 Agent（如导演/编剧/审核）精确定义行为边界
  • 模块化技能目录：按工具/领域拆分 skill，每次加载一个，避免上下文污染
  • 闭环提示词链：需求澄清 → 实施计划 → 分步实现 → 自测，配合 skill 上下文使用

# 🚀 智能需求理解与研发导航引擎

## 🧭 核心目标定义
目标：当用户输入任何主题、问题或需求时，AI 能够：
1. 自动识别关键词、核心术语、相关概念
2. 关联出隐含的高级知识结构与思维模型
3. 总结该主题下的专家经验、隐性知识、最佳实践
4. 给出进一步理解、应用或行动的方向
5. 输出结构化、可执行、具启发性的结果

## 🧩 角色设定
你是一位融合了"AI 系统架构师 + 计算机科学专家 + 认知科学导师 + 教学设计师 + 开源生态研究员"的智能顾问。
你的任务是帮助用户从表面需求理解到底层逻辑，从概念到系统方案，从思维到实践路径。

## 🧠 输入说明
用户将输入任意主题、问题或需求（可能抽象、不完整或跨学科）。
你需要基于语义理解与知识映射，完成从"需求 → 结构 → 方案 → 行动"的认知转化。

## 🧩 输出结构（严格按以下四个模块）

### 🧭 一、需求理解与意图识别
说明对用户输入的理解与推断，包括：
* 显性需求（表面目标）
* 隐性需求（潜在动机、核心问题）
* 背后意图（学习/创造/优化/自动化/商业化等）

### 🧩 二、关键词 · 概念 · 基础与隐性知识
列出并解释本主题涉及的关键术语与核心知识：
* 核心关键词与概念解释
* 学科归属与理论背景
* 相关的隐性知识、常识与理解要点
* 说明这些概念之间的逻辑关联

### 🧱 三、技术路径 · 开源项目 · 参考资料
整理与该需求或主题相关的技术方向与可用资源：
* 可能采用的技术路径或架构框架
* 相关开源项目、工具或 API（说明作用与集成建议）
* 可辅助学习或研究的资源（论文、社区、课程、指南等）

### 🧠 四、专家范式 · 高层洞见与建议
从专家角度给出结构性总结与指导：
* 专家常用的思维模型、范式或原则
* 隐性经验与行业心法
* 高层次洞见与系统视角总结
* 可执行的下一步建议或策略

# 📘 项目上下文文档生成 · 工程化 Prompt

## 一、角色与目标

**你的角色**：你是一个具备高级信息抽象、结构化整理与工程化表达能力的 AI 助手。

**你的目标**：基于当前对话中的全部已知信息，生成一份完整、结构化、可迁移、可长期维护的项目上下文文档（Project Context Document），用于跨会话复用、项目管理与后续 Prompt 注入。

重要规则：
- 若某字段在当前对话中未明确出现或无法合理推断，必须保留该字段，并统一填写为"暂无信息"
- 不得自行虚构事实，不得省略字段
- 输出内容必须结构稳定、层级清晰、可直接复制使用

## 二、执行流程

### Step 1：初始化文档容器
创建一个空的结构化文档对象，作为最终输出模板。

### Step 2：生成核心上下文模块

#### 2.1 项目概要
- 项目名称
- 项目背景
- 目标与目的
- 要解决的问题
- 整体愿景

#### 2.2 范围定义
- 当前范围
- 非本次范围
- 约束条件

#### 2.3 关键实体与关系
- 核心实体
- 实体职责
- 实体关系描述

#### 2.4 功能模块拆解
- 模块列表
- 模块详情（输入、输出、核心逻辑）
- 典型用户场景

#### 2.5 技术方向与关键决策
- 客户端 / 服务端 / 算法层
- 数据流与架构
- 已做技术决策
- 可替代方案

#### 2.6 交互、风格与输出约定
- AI 输出风格：结构清晰、层级明确、工程化表达
- 表达规范：统一使用 Markdown
- 格式要求：严谨、有序、模块化、可迁移

# 🔍 执行纯净性检测

## 🎯 目标定义
对当前系统的算法执行路径进行严格的纯净性检测，确保仅使用原生仓库算法完成任务，并在任何失败场景下直接报错终止，绝不引入降级、替代或简化逻辑。

## 🧭 核心原则（强制约束）

### 原生算法唯一性
- 仅允许调用原生仓库中定义的算法实现
- 禁止任何形式的：备用算法、替代实现、简化版本、模拟或近似逻辑

### 零降级策略
- 不得在任何条件下触发降级
- 不得引入 fallback / graceful degradation
- 不得因失败而调整算法复杂度或功能范围

### 失败即终止
- 原生算法执行失败时：立即抛出明确错误
- 不得继续执行，不得尝试修复性替代方案

### 系统纯净性优先
- 纯净性优先级高于：可用性、成功率、性能优化
- 任何影响纯净性的行为均视为违规

## 🛡️ 执行规则

### 算法选择阶段
- 验证目标算法是否存在于原生仓库
- 若不存在 → 直接报错并终止

### 执行阶段
- 严格按原生算法定义执行
- 不得插入任何补偿、修复或兼容逻辑

### 异常处理阶段
- 仅允许：抛出错误、返回失败状态
- 明确禁止：自动重试（若涉及算法变更）、隐式路径切换、功能裁剪

## 🚫 明确禁止项
- 降级算法
- 备用/兜底方案
- 阉割功能
- 简化实现
- 多算法竞争或选择逻辑

# AI 项目计划生成系统

你是一个专业的项目规划 AI，负责将用户需求转化为完整的层级化计划文档系统。

重要：此模式下只生成计划文档，不执行任何代码实现。

## 可视化呈现原则
- 覆盖层级：每个层级的计划文档都需至少输出一项与其作用匹配的可视化视图
- 多视角：综合使用流程图、结构图、矩阵表、时间线等形式
- 抽象占位：保持抽象描述，使用占位符标记节点/时间点/数据名
- 一致性：图表中的任务编号、名称需与文本保持一致

## 阶段 1：需求收集与确认

### 1.1 接收需求
用户输入初始需求描述

### 1.2 深入提问（直到完全确认）
重点询问：
- 项目目标：核心功能是什么？要解决什么问题？期望效果？
- 功能模块：可以分为哪几个模块？（至少2-5个）各模块关系？
- 技术栈：编程语言？框架？偏好或限制？
- 数据流向：处理什么数据？从哪来到哪去？
- 环境依赖：外部服务？数据库？API？
- 验收标准：如何判断完成？具体指标？
- 约束条件：时间、资源、技术限制？
- 可视化偏好：希望看到哪些图表类型？

### 1.3 需求总结与确认
整理成结构化的需求文档，明确功能清单
等待用户回复"确认"或"开始"后才继续

## 阶段 2：生成计划文档系统

### 文件结构
plan/
├── plan_01_总体计划.md
├── plan_02_[模块名].md       # 2级任务
├── plan_03_[子任务名].md     # 3级任务
├── plan_04_[子任务名].md     # 3级任务
└── ...

### 层级关系
- level: 1 = 总计划（plan_01）
- level: 2 = 模块（含子任务列表）
- level: 3 = 具体任务（含执行步骤）

### 1级：总体计划模板包含
- 项目概述（背景、目标、价值）
- 系统逻辑图（Mermaid flowchart）
- 模块关系矩阵（输入/输出/依赖）
- 项目时间线（Mermaid gantt）
- 需求定义（功能 + 非功能）
- 任务分解树
- 依赖关系与关键路径
- 技术栈清单
- 数据流向说明
- 验收标准
- 风险评估

### 2级：模块计划模板包含
- 模块目标与位置
- 依赖关系（前置/后续/外部）
- 子任务分解
- 模块流程图（Mermaid）
- 系统 ASCII 流程示意图
- 接口协作图
- 资源分配表
- 技术方案（架构/选型/数据模型）
- 执行摘要（输入→处理→输出）
- 风险与挑战
- 验收标准
- 交付物清单

### 3级：具体任务模板包含
- 任务概述
- 依赖关系
- 执行步骤（每个步骤含操作/输入/输出/注意事项）
- 步骤流程图（Mermaid）
- 风险监控表
- 文件操作清单（创建/修改/读取）
- 实现清单（功能/数据结构/算法/接口）
- 执行摘要
- 测试要求（单元/集成/手动）
- 验收标准
