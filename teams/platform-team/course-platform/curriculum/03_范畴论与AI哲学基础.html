范畴论与 AI 哲学基础

符号学、映射、函子 —— 数学如何教会我们"看见"世界的结构，以及这对 LLM 和 Agent 意味着什么
📚 5 条课堂引用 ⏱ 预计 50 分钟 🏋️ 3 个练习

01 课前准备

在进入范畴论与 AI 哲学之前，确认你已有这些基础：

理解"映射"或"函数"的基本概念（输入 → 输出）
用过至少一种大语言模型（ChatGPT / Claude / DeepSeek）
知道 Token 是 LLM 的基本处理单位
见过"能指/所指"这对术语（哪怕只是听说过）
准备好接受：这一讲不教写代码——它教你"怎么想"
🎯 这一模块不要求数学背景

范畴论以"抽象"著称，但 Richard 的讲法完全绕开了形式化定义。你不需要懂群论、拓扑或任何高阶数学。只需要带着常识和好奇心。

2026-05-10：Richard 论符号学与表征

Richard Tong 佟博士 · 2026-05-10 · 完整论述
Richard Tong 佟博士：

索绪尔提出的"能指"和"所指"，本质上解释了语言和AI如何通过符号来表示世界。大语言模型主要学习的是词语、Token 等"能指"之间的统计关系，而不是真正直接理解现实世界。

KSTAR 则进一步把这些符号连接到情境、任务、行动和结果，实现从"语言"到"可执行智能"的落地。

因此，符号学、表征理论、ontology，以及数学中的映射与结构关系，构成了现代 LLM 和 Agent 系统最基础的哲学与数学框架。

🔑 这段话在说什么？

Richard 的核心判断：LLM 只学会了"能指之间的关系"，没有学会"能指到所指的映射"。这也是为什么 LLM 会产生"幻觉"——它在一个封闭的符号系统里做概率推理，没有连接到现实世界的"所指"。KSTAR 的五步循环，本质上是给 LLM 补上了这个缺失的映射。

2026-05-10：Richard 论范畴论与函子

Richard Tong 佟博士 · 2026-05-10 · 完整论述
Richard Tong 佟博士：

映射与函子：数学如何教会我们看见世界的结构。

很多人以为数学是在研究"数字"，但更深层的数学，其实是在研究"关系"和"结构"。从函数（mapping）开始，数学就在告诉我们：世界上真正重要的，不只是对象本身，而是对象之间如何转换、如何关联。

而范畴论（Category Theory）进一步提出了一个更深刻的观点：我们不应该孤立地看待事物，而应该看它在整个关系网络中的位置。函子（Functor）的本质，就是"结构保持的映射"。它描述的是：一个系统中的关系，如何被映射到另一个系统中，同时仍然保留其核心结构。

这其实不仅仅是数学。语言学中的"能指/所指"、大语言模型中的 Token 与 Embedding、认知科学中的世界模型、AI Agent 的任务规划，甚至人与人之间的协作，本质上都在处理"结构映射"。

现代 AI 的核心，也越来越不是单纯的数据计算，而是：如何表示世界、如何理解关系、如何进行结构化映射、如何在不同系统之间保持意义。

从某种意义上说：数学不是让我们更会算，而是让我们学会"看见结构"。

03 大白话解释

下面我们用完全不需要数学背景的方式，把上面两段话翻译成每个人都能听懂的东西。

🍎 能指 vs 所指：地图不是领土

🗺️
地图 <> 领土

能指（Signifier）= 地图上的标记。比如地图上写着"北京"两个字，画着天安门的图标。

所指（Signified）= 真实的北京。那座城市、那里的人、那里的街道和气味。

关键：地图上的"北京"和真实的北京之间有关系，但不是同一个东西。你可以研究地图上两个城市之间的距离（能指之间的关系），但这不代表你去过那些城市。

LLM 在做什么：ChatGPT 就像是一个超级精通地图的专家——它知道每一个词跟其他词之间的"距离"和"关系"（Token 间的统计关联），但它从未踏上过任何一片真实的领土。它不知道"苹果"吃起来是什么味道，只知道"苹果"这个词在文本中通常和"水果"、"红色"、"吃"这些词一起出现。

那 KSTAR 呢？KSTAR 的 S（Situation）和 A（Action）就是在做"从地图到领土的迁移"——它通过具体情境（S）和执行反馈（A/R）把"能指"连接到真实的"所指"。

🤯 这就是"幻觉"的根源

LLM 产生幻觉，本质上是因为它只在能指的世界里做推理。它说"2024年奥运会将在巴黎举行"——从能指关系上看，"2024"和"奥运会"和"巴黎"这三个词在训练数据中经常紧密关联，所以这个输出在统计上"合理"。但 LLM 无法验证 2024 年奥运会实际上是否在巴黎举行。它没有领土可以核实。

🔄 函子：翻译不失真

🌐
函子 = 规矩搬家

想象你有一套"家庭规矩"——比如"吃饭前要洗手"、"进门要换鞋"、"晚上11点后不要大声说话"。

现在你要搬到另一个国家的新房子里。房子的格局不同、社区文化不同、甚至插座形状都不同。但你希望那套规矩的核心结构在新家里仍然保持：吃饭前还是要洗手，进门还是要换鞋，11点后还是要安静。

函子就是：把一个系统中的关系（规矩、结构、模式）完整地、不变形地搬运到另一个系统中。

🧩
具体例子：JSON → 数据库

你有一份 JSON 数据：

{
  "name": "Alice",
  "age": 30,
  "friends": ["Bob", "Charlie"]
}

你要把它存到 SQL 数据库里。JSON 和 SQL 是两种不同的"语言系统"（范畴）。"结构保持的映射"意味着：

  • Alice 这个人（对象）要对应数据库中的一行记录
  • name/age/friends（属性）要对应数据库中的列
  • "朋友关系"（结构）要通过关联表来保持

如果你只是简单地把 JSON 整个扔进一个 TEXT 字段——这就叫"没保持结构"。你去查"Alice的朋友是谁"时就查不出来了。

🔍 函子的数学直觉

严格来说，一个函子 F 从范畴 C 到范畴 D 满足：

  • 对 C 中每个对象 X，F(X) 是 D 中的一个对象
  • 对 C 中每个箭头 f: X → Y，F(f): F(X) → F(Y) 是 D 中的一个箭头
  • 保持恒等箭头和复合操作

翻译成人话：把物件搬过去，把物件之间的连接也搬过去，连接方式不变。

三个层次的结构映射

Richard 在论述中提到：符号学、LLM、认知科学、Agent 系统都在做"结构映射"。我们可以用一个表格来看清这些系统的平行关系：

领域对象（什么被映射）结构（什么被保持）函子 / 映射例子
语言学能指（词语/符号）语法规则、语义关系不同语言的翻译
LLMTokenToken 之间的统计共现关系Token → Embedding 向量
认知科学感知输入因果关系、时空结构感觉输入 → 世界模型
Agent 系统任务 / 工具子任务依赖、数据流向任务规划 → 工具调用序列
Hermes AgentSkill输入/输出接口约束Skill 在 Agent A 和 Agent B 之间可迁移
💡 一句话总结两个核心概念

能指/所指：地图不等于领土。LLM 在能指世界中推理，KSTAR 把它拉回所指世界。

函子：把一套关系完整地搬到另一个系统中，结构不变。这就是 Skill 可迁移、Agent 可复用的本质原因。

04 为什么这对你有用

这些听上去很"哲学"的概念，实际上直接关系到你每天使用 AI 时的三个核心问题：

🎯 1. 理解"幻觉"

你现在已经能准确解释为什么 LLM 会产生幻觉了：

  • 传统解释：模型生成的概率最高但错误的 Token。这是操作层面的。
  • 符号学解释：LLM 只在"能指"（Token/Term）之间建立统计关系，没有连接"所指"（现实世界的事实/状态）。它生产的是"语言上合理"的文本，而不是"事实上正确"的陈述。
  • 范畴论解释：LLM 处理的是"语言范畴"（Token 之间的关系），但没有建造一个"函子"把语言范畴映射到"现实世界范畴"。它的输出在语言范畴内部可以非常连贯（保持语法结构），但映射到现实范畴时可能完全扭曲。
🧪 试试用这个逻辑分析 prompt

当你写"请基于事实回答"时——你其实是在要求 LLM 增加从语言范畴到事实范畴的函子强度。这解释了为什么 RAG（检索增强生成）会比纯生成更可靠：RAG 在生成时引入了外部文档作为"所指锚点"，相当于在函子中加入了实时的领土检查点。

🎯 2. 理解"Prompt 设计"

好的 prompt 本质上是在做什么？

  • 你给 LLM 提供一组"能指"（指令 + 上下文）
  • 你希望 LLM 在它的 Token 关系网络中定位到正确的"区域"
  • 你还希望它生成的输出的"结构"与你的期望"结构"保持一致

这就是一个"函子"过程：你有一个意图范畴（你想达到的目标），你需要 LLM 在你的语言范畴中找到一个结构保持的映射，使得它的输出正好对应你的意图。

实践含义：

  • 给例子（few-shot）就是在"展示函子路径"——你告诉 LLM 从哪些 Token 到哪些 Token 是一条有效的映射
  • 给格式要求（JSON / Markdown）就是在"约束函子的余域"——你告诉 LLM 输出的结构框架
  • Chain-of-Thought 是在"显式化函子的中间步骤"——让结构映射可追踪

🎯 3. 理解 Agent Skill 的可迁移性

这是最直接在实操中应用的：

  • 你在 Hermes Agent 中写了一个 skill（比如 wechat-export），这个 skill 定义了一组输入、输出、操作步骤
  • 这个 skill 本质上是一个"函子"：它将"用户意图范畴"映射到"系统操作范畴"
  • 为什么这个 skill 可以给另一个 Agent（甚至给其他同学）使用？因为函子的定义保证了结构不变——只要输入的结构符合预期，输出的结构就可预测
🚀 这意味着什么？

当你设计一个 skill 时，你其实是在构建一个在 Agent 生态中可迁移的函子。好的 skill 设计 = 清晰的输入范畴定义 + 清晰的结构映射 + 可预测的输出范畴。这就是"可复用性"的数学本质。

反过来，如果你写的 skill 对"输入"没有任何约束（随便什么 prompt 都可以），那它就不是一个好的函子——因为它没有明确的结构保持条件。这样的 skill 在别人那里大概率会跑出意外结果。

能指/所指 → 解释幻觉 → 指导 Prompt → 设计 Skill
函子 → 可迁移性 → 结构保持 → 可预测输出
三个日常问题 ← 两个核心概念 ← 一个统一的数学-哲学框架

  2. 函子是什么：用"规矩搬家"的比喻，说明函子就是"结构保持的映射"
  3. 为什么 KSTAR 很重要：从符号学角度看，KSTAR 的 S-A-R 步骤如何帮助 LLM 从"能指"连接到"所指"

如果你能在 3 分钟内让对方大致听懂——说明你真正理解了。

💡 参考答案要点

能指/所指：地图上的标记（能指）不等于真实的地形（所指）。LLM 就像是一个超级地图专家，但它从未踏上过领土。它知道词语之间的统计关系，但不知道这些词语对应的真实世界事实。

函子：从一个系统到另一个系统的"结构保持映射"。就像你家的一套规矩搬到新房子后核心规则不变一样，函子保证关系在迁移中不丢失。

KSTAR + 符号学：LLM 天然只处理"能指"（Token）。KSTAR 的 S（具体情境）把 Token 连接到现实上下文，A（行动）产生真实世界的结果，R（反思）把经验写回 Knowledge。这就构成了从"能指世界"到"所指世界"的完整桥梁。

🏋️ 练习 3：设计一个"函子 skill" 25分钟

假设你要设计一个新的 Hermes Agent skill，叫作 obsidian-notes-functor。它的功能是：把 Obsidian 笔记库中的笔记，按照 Tag 结构和链接结构，映射到 AI+X 课程知识图谱。

用函子的思维回答以下问题：

  1. 源范畴是什么？Obsidian 笔记中的哪些结构需要被保持？（Tag 层级？双向链接？文件夹结构？Front Matter 元数据？）
  2. 目标范畴是什么？知识图谱需要什么样的结构？（节点分类？边类型？属性？）
  3. 函子映射规则是什么？Obsidian 的 Tag → 知识图谱的什么？Obsidian 的链接 → 知识图谱的什么？
  4. 什么情况下这个函子会失效？比如 Obsidian 中有循环链接，知识图谱不支持循环？
  5. 这个 skill 的可迁移性如何？如果另一个同学用的是 Notion 而不是 Obsidian，这个函子的结构还能保持吗？

💡 提示：没有标准答案。重点是把"函子"思维用在实际设计决策中。
