# C09 · 真实项目交付

> 难度：L3（高级） | 类型：产品型 | 预估耗时：2-3周

## 📋 挑战概述

将前面学到的所有技能整合起来，为一个**真实需求方**交付一个可部署的产品。这是从"学习者"到"交付者"的转变——你的产出有真正的用户在使用。

> **核心理念**：真实项目不是"做作业"，而是在真实需求中学习、在真实挑战中迭代、在真实反馈中成长。

### 前置要求
- 完成C05（单智能体系统开发）或具备同等能力
- 有一个真实的需求方（老师、企业、同学、社团等）
- 承诺每周至少投入10小时

## 🎯 学习目标

完成本挑战后，你将能够：

1. 从需求方获取需求并转化为技术方案
2. 制定项目计划并在2-3周内交付
3. 在AI辅助下独立完成从设计到部署的全流程
4. 向非技术人员展示和讲解你的产品

## 📦 交付物

| 产物 | 说明 | 最低标准 |
|------|------|---------|
| ① 项目任务书 | 需求文档、技术方案、项目计划 | 需求方确认过 |
| ② 可部署产品 | 上线可用的产品 | 需求方能实际使用 |
| ③ 用户手册 | 如何使用你的产品 | 非技术人员能看懂 |
| ④ 演示视频（3分钟） | 本人+作品出镜的展示 | 展示核心功能和工作原理 |
| ⑤ AAR复盘 | 课后复盘 | 包含完整项目交付的反思 |

## 🧩 挑战步骤

### Step 1: 确定项目（2天）

**找需求**：可以从以下渠道获取真实需求：

| 来源 | 示例 |
|------|------|
| 学校老师 | 课程管理系统、作业批改工具 |
| 同学/社团 | 活动报名系统、知识库 |
| 企业/实习 | 内部工具、自动化流程 |
| 自己 | 个人作品集、博客 |

**你的任务**：和需求方至少沟通2次，用AI辅助整理需求文档。

> "我和[需求方]沟通后，他们需要的是一个[产品描述]。核心功能包括：1. ... 2. ... 3. ... 请帮我基于这些需求生成一份项目任务书模板。"

### Step 2: MVP开发（1周）

**最小可行产品（MVP）原则**：
- 只做最核心的2-3个功能
- 先跑通再优化
- 每周要给需求方看一次进展

**用AI加速开发**：

| 开发阶段 | AI可以帮你 |
|---------|-----------|
| 技术选型 | 推荐技术栈、对比优劣 |
| 代码生成 | 生成功能模块代码 |
| 调试 | 分析报错、修复bug |
| 测试 | 生成测试用例 |
| 文档 | 编写使用说明 |

### Step 3: 部署上线（3天）

选择部署方式：

| 方式 | 适用 | 难度 |
|------|------|------|
| GitHub Pages | 静态网站 | ⭐ |
| Vercel/Netlify | Web应用 | ⭐⭐ |
| 本地服务器 | 企业内部工具 | ⭐⭐⭐ |
| Railway/Fly.io | 后端服务 | ⭐⭐⭐ |

> "我的项目已经开发完成，是一个[技术栈描述]。请帮我写一份Vercel部署指南，包括环境变量配置和构建命令设置。"

### Step 4: 验收与展示（2天）

- 给需求方演示产品
- 收集反馈，修复关键问题
- 录制3分钟演示视频
- 准备路演PPT

### Step 5: AAR复盘

## 📐 评分标准（Rubric）

| 维度 | 通过 (60%) | 良好 (80%) | 优秀 (100%) |
|------|-----------|-----------|-----------|
| **需求落地** | 基本功能实现 | 功能匹配需求 | 超出需求方预期 |
| **产品完成度** | 能运行 | 核心功能完整 | 有完善UI/UX和错误处理 |
| **部署状态** | 本地可运行 | 已部署上线 | 有域名/持续更新 |
| **用户反馈** | 需求方看到了 | 需求方试用过并认可 | 需求方愿意持续使用 |
| **交付质量** | 完成了交付物 | 交付物完整可用 | 有完整的项目复盘和技术文档 |

## 💡 提示与常见坑

- ✅ **MVP原则**——"做得快比做得多重要"，先有一个能用的版本
- ✅ **承诺少一点，交付多一点**——给需求方超出预期是加分
- 🚫 **不要等全部做完才给需求方看**——每3天同步一次
- 🚫 **不要接做不完的项目**——2-3周内无法完成的就不接
- ✅ **把AI当你的"全栈团队"**——前端、后端、测试、文档，AI都能帮你
